# 전략 제안: (worker-3) 평가/검증 전략

## 1. phase6_eval.py 5-metric gate의 충분성 검토
**결론: 도메인 적응(Style Adaptation) 평가에는 "불충분"함.**

- **이유:** 현재 5개 지표(`bigram_jsd`, `length_kl`, `digit_density`, `english_density`, `mauve`)는 통계적 분포와 표면적 텍스트 특성만 측정함. 
- **한계:** 
    - 도메인 특화 용어(TC, 밀빵, 케어 등)의 사용 정확도와 빈도를 직접 평가하지 않음.
    - 문체(말투, 존비어 사용 비중)의 질적 변화를 포착하기 어려움.
- **보완 제안:** 
    - **Domain Keyword Alignment (DKA):** `refinement_loop.py`의 Phase 1a 로직을 gate에 통합. 핵심 20개 용어의 원천 대비 출현 비중(ratio)이 0.5 이상인지 확인.
    - **Tone Distribution Match:** 반말/존댓말 비중이 원천 데이터의 ±15% 이내인지 측정 (Phase 1b 로직).

## 2. bigram JSD PASS 기준 설정
- **베이스라인 (0.019):** 원천 데이터 간(raw-vs-raw)의 자연적 변동치.
- **현재 Threshold (0.15):** 너무 완만함. 일반적인 한국어 모델 수준만 유지해도 통과 가능.
- **제안 기준:**
    - **PASS (준수):** ≤ 0.08 (Stretch goal 수준). 베이스라인의 약 4배 수준으로, 이 수치를 넘어가면 '기본 모델의 문체'가 지배적이라고 판단.
    - **GOLD (우수):** ≤ 0.04. 베이스라인의 2배 이내로, 스타일 전이가 성공적으로 이루어졌음을 의미.

## 3. Blind Eval 없이 가능한 로컬 검증 전략
1. **Perplexity (PPL) 측정:** `val_set.v2.jsonl`에 대한 Loss/PPL 측정. Base 모델 대비 유의미한 하락이 있어야 함.
2. **Keyword Recall Test:** SFT용으로 준비된 Instruction을 입력했을 때, 답변에 핵심 도메인 용어(TC, 갯수 등)가 적절한 문맥에서 생성되는지 Recall 측정.
3. **LLM-as-a-Judge (Proxy):** GPT-4o나 Claude 3.5 Sonnet을 활용하여 "달빛알바 커뮤니티 원주민스러움"을 1~10점으로 평가. 
4. **N-gram Overlap (Jaccard Similarity):** 원천 데이터의 고빈도 문장 구조와의 유사도 측정.

## 4. 용어 빈도 GAP이 생성 품질에 미치는 영향
- **TC GAP (0.10 → 0.06)의 치명성:** 'TC'는 해당 도메인의 핵심 비즈니스 로직(비용/수익)을 담고 있음. 이 용어의 노출이 극도로 낮아지면 모델은 해당 개념을 표준어(봉사료, 수수료)로 대체하거나 잘못된 문맥에서 사용할 가능성이 높음.
- **환각(Hallucination) 위험:** 도메인 용어의 학습 빈도가 낮으면 Qwen3-Base가 알고 있는 일반적인 한국어 의미로 회귀하여, 커뮤니티 특유의 은어적 의미를 상실함.
- **체감 품질(Vibe) 저하:** 용어 빈도 불균형은 '말은 되는데 우리 식구 말투는 아니다'라는 이질감을 유발하여, 최종 사용자의 수용성을 크게 떨어뜨림. TC/밀빵 등 핵심어의 소실은 CPT의 존재 목적(문체 적응)을 부정하는 결과로 이어짐.
