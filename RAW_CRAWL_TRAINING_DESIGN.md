## 현재 파이프라인의 장점

- raw와 스냅샷(`cpt_corpus.jsonl`, `sft_pairs_v2.jsonl`)의 연결률은 높다.
- SFT comment pair 상당수는 실제 raw reply chain으로 역추적된다.
- 도메인 은어와 구어체는 데이터 자체에 충분히 존재한다.
- 옵시디언 export에도 `topic_label`, `slang_tokens`, `honorific`, `comment_count`,
  `분석 노트/감정 톤/언어적 특징` 같은 보조 메타가 있어 raw 분석 결과를 교차검증할 수 있다.

## 현재 파이프라인의 한계

### 1. CPT가 구조 정보를 버린다

`train_cpt.py`는 최종적으로 `text`만 학습한다.

- `kind` 사용 안 함
- `title` 사용 안 함
- `boardName` 사용 안 함
- 댓글의 parent-child 구조 사용 안 함

즉 말투는 배우지만, "어떤 상황에서 어떤 텍스트가 붙는지"는 약해진다.

### 2. comment SFT가 post 문맥을 거의 안 본다

현재 `sft_pairs_v2.jsonl` comment pair는 주로

- 부모 댓글 텍스트
- 자식 댓글 텍스트

위주로 학습된다. 하지만 실제 댓글의 문맥에는

- 원글 제목
- 원글 본문
- thread depth
- 앞선 댓글 흐름

이 모두 영향을 준다.

### 3. 광고/운영진 텍스트 오염 가능성

광고성 댓글이 raw에서 매우 많기 때문에, 필터가 약하면

- 홍보 전화번호
- 카톡 ID
- 정형화된 케어 문구
- 영업진 말투

를 사용자 말투와 함께 학습할 수 있다.

### 4. eval이 충분히 thread-aware 하지 않다

post/comment를 같이 보도록 보강 중이지만,
"원글 + 부모 댓글 + 답글" 구조 전체를 평가하는 전용 루프는 아직 없다.

## 권장 학습 설계

### A. raw 정규화 단계

- `post_id`, `title`, `content`, `boardName`, `date`, `crawledAt` 보존
- 댓글 prefix 정규화:
  - `[2]`
  - `[2-1]`
  - `비회원 [2-1]`
  - `작성자 [2-3]`
- `comment_key`, `parent_comment_key`, `depth` 생성
- 광고/삭제/블라인드 텍스트는 별도 태그 부여

### B. CPT 설계

`text` 단일 컬럼만 쓰지 말고 최소한 아래 두 계열로 나눈다.

1. 게시글 코퍼스
- `제목 + 본문`

2. 댓글 코퍼스
- 정규화된 댓글 본문

권장:
- 광고성/삭제/블라인드 댓글 제외
- 고빈도 중복 홍보 댓글 dedupe
- 최근 데이터 가중치 강화

### C. SFT 설계

#### 게시글 SFT

- 제목 -> 본문
- 주제 -> 짧은 게시글
- 첫 문장 -> 이어쓰기

#### 댓글 SFT

현재보다 richer context로 바꾼다.

입력 예시:
- 게시글 제목
- 게시글 본문 요약 또는 앞부분
- 부모 댓글
- thread depth

출력:
- 실제 자식 댓글

이렇게 해야 댓글이 "원글과 무관한 짧은 반응"이 아니라
실제 thread 문맥에 맞는 답글로 학습된다.

추가로 root comment와 reply comment를 분리해서 다뤄야 한다.

- root comment: 원글에 바로 다는 첫 댓글
- reply comment: 다른 댓글을 받고 이어지는 답글

이 둘은 길이, 감정 톤, 초성 사용 비율이 다르다.

### D. Eval 설계

eval은 최소 3축으로 분리한다.

1. 게시글 realism
- AI 게시글 vs 실제 게시글

2. 댓글 realism
- AI 답글 vs 실제 답글

3. thread coherence
- `원글/부모 댓글/답글` 묶음으로 judge 평가

추가로 rule-based sanity check 필요:
- 전화번호/카톡/텔레그램 포함률
- 과도한 정중체 비율
- 슬랭 marker 비율
- 초성 marker 비율
- 질문형/감탄형/말줄임표 비율
- 길이 분포 이탈

## 진행 우선순위

1. raw profiler를 기준으로 필터 규칙 고정
2. raw -> normalized thread dataset 생성
3. thread-aware comment SFT 데이터 재생성
4. post/comment/thread 분리 eval 재실행

## 현재 판단

현재 파이프라인은 "도메인 말투와 주제를 대체로 학습"하는 수준까지는 맞다.
하지만 "실제 댓글이 달리는 문맥과 흐름까지 전방향적으로 구현"했다고 말하긴 아직 이르다.

핵심 부족분은 두 가지다.

- raw 노이즈 필터링이 구조적으로 고정돼 있지 않다.
- comment generation이 full thread context를 아직 쓰지 않는다.
