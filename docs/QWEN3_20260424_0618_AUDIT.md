# Qwen3 20260424-0618 Audit

Audit time: 2026-04-27 KST

## Verdict

The training direction was basically appropriate for the project goal:

- Base model: `Qwen/Qwen3-8B-Base`
- Method: QLoRA CPT -> merge -> SFT -> deterministic eval
- Target: Korean community/style adaptation from curated crawl snapshots

The failure was not simply that the budget was too small. The stronger diagnosis is:

1. The runtime and budget assumptions were too optimistic for a full CPT + SFT chain.
2. The pipeline did not preserve enough final artifacts and logs to prove completion.
3. Retry / smoke / promotion gates were weak, so paid GPU time was spent before the run was cheap to validate.

- However, completion cannot be accepted without preserved SFT adapter, train logs, and eval artifacts.

## RunPod Cost Judgment

Observed CPT speed from the run was about `18.43 sec/step`.

Current dataset / recipe estimate:

- CPT: `5775` steps, about `29.57h`, about `$23.36`
- SFT: about `14440` steps, about `73.92h`, about `$58.40`
- Train total before eval/retries/storage: about `103.49h`, about `$81.76`

Interpretation:

- A `$90` budget is barely enough for one clean full run.
- It is not enough for an autonomous retry loop with evaluation, failures, storage, and re-runs.
- So the practical problem was not only insufficient estimate. The pipeline also needed cheap smoke checks and stronger artifact gates before spending on the full chain.

## Promotion Rule

Do not upload or announce a final HF model until all of these are true:

1. Local strict verification has severe findings `0`.
2. Smoke train run completes with `done_ok`.
3. HF CPT repo contains an adapter and complete checkpoint metadata.
4. HF SFT repo contains `sft-lora/adapter_model.safetensors` or root `adapter_model.safetensors`.
5. Phase6 eval artifact branch exists and `metrics.json` is reviewed.
6. A full run budget is allocated with retry margin, not just the nominal one-pass estimate.
