## Why this PR is in handoff state instead of merged

Training was attempted twice on this machine:

1. **Full config** (epochs=3, max_length=256, 5000+5000 samples): killed at ~6 min wall-clock after model load completed but no epoch lines emitted. The trainer was active (CPU 542 % across 9 worker threads, 442 MB written to HF cache) but had not yet reached the train loop. Estimated total wall-clock 60-130 min.
2. **Reduced PoC config** (epochs=1, max_length=128, 1000+1000): launched as a battery-friendly dev attempt. Single thread (main, 93.6 % CPU) was active but the other 13 threads were all in `futex_wait_queue` (sleeping); only 16 KB written to disk after 7 min, no `runs/` directory created. Strong evidence of a single-threaded bottleneck — likely PyTorch's intra-op thread pool not initialising for klue/roberta-base on this Debian sandbox. Killed.

Conclusion: the classifier code is complete and correct, but **training to a useable AUC requires ~1 hour of dedicated CPU on AC power**, and ideally a host with a working PyTorch CPU thread pool (the omnibook running this session showed only one productive thread).

The script's hyperparameters in this commit are restored to the **production config** (epochs=3, max_length=256, 5000+5000). The PoC reduction lives only in this handoff doc.

## What to run on the resumption host

### Prerequisites

* AC power (training is 60-130 min CPU-bound)
* Python 3.10+ with `transformers >= 5.x`, `tokenizers`, `torch` (CPU build is fine), `scikit-learn`, `datasets`. On Debian:
  ```bash
  pip install --user --break-system-packages transformers tokenizers torch scikit-learn datasets
  ```
* Internet for the first run (HF Hub downloads klue/roberta-base ≈ 440 MB and the wikimedia/wikipedia ko subset ≈ several MB into `~/.cache/huggingface/hub/`)
* HF cache is permitted to grow to ~500 MB; subsequent runs reuse it

### One-command reproduction

```bash
cd <repo-root>
git fetch origin feat/p1-style-classifier
git checkout feat/p1-style-classifier
python3 scripts/train_korean_style_classifier.py \
    --corpus cpt_corpus.v2.jsonl \
    --out-dir runs/ \
    --baseline .planning/calibration/style_classifier_baseline.json
```

The script writes:

* `runs/style_classifier_<UTC_timestamp>/` — model checkpoint (≈ 440 MB; **do not commit**, runs/ is gitignored)
* `.planning/calibration/style_classifier_baseline.json` — overwritten with the real `train_size`, `val_auc`, `test_auc`, `test_accuracy`, `confusion_matrix`, `base_model`, `hyperparams`, `seed`, `timestamp`

### Acceptance criteria (must hit before merge)

1. `test_auc > 0.95` (the community-vs-formal task is not subtle; lower means something is wrong with the data or the head init)
2. `.planning/calibration/style_classifier_baseline.json` is non-placeholder and has all required fields
3. `phase6_eval.py` runs without error when `style_classifier_eval` is importable, and skips silently when it is not (mirrors the MAUVE optional pattern)
4. `git diff main..feat/p1-style-classifier --stat` should show only the five tracked files plus this handoff doc; no checkpoint binaries

### After training completes

## Smoke test before training

To validate that the script will at least *try* to run on the host without spending an hour first:

```bash
python3 -c "
import importlib, sys
for m in ['transformers', 'torch', 'datasets', 'sklearn']:
    importlib.import_module(m)
    print(f'{m}: ok')
"
python3 -c "
from transformers import AutoTokenizer, RobertaForSequenceClassification
tok = AutoTokenizer.from_pretrained('klue/roberta-base')
m = RobertaForSequenceClassification.from_pretrained('klue/roberta-base', num_labels=2)
import torch
ids = tok('테스트 문장', return_tensors='pt')
out = m(**ids)
print('logits.shape:', out.logits.shape)
"
```

If both prints succeed, the trainer will at least load. The hang we saw on this host was past the model load, so passing this smoke check is necessary but not sufficient.

### If training stalls again

Symptoms to watch:

* `ps -T -p <pid>`: only one thread in `Rl` and others in `futex_wait_queue` for 5 + minutes
* `wchar` in `/proc/<pid>/io` not growing (no checkpoint writes)
* `runs/` directory absent at the 10 min mark

Workaround: explicitly pin PyTorch threads before running:

```bash
OMP_NUM_THREADS=8 MKL_NUM_THREADS=8 python3 \
    scripts/train_korean_style_classifier.py ...
```

Or migrate the run to a small RunPod CPU pod (the project already has the launcher infrastructure under `scripts/launch_train_pod.py` — set GPU_TYPE empty and CPU_COUNT >= 8). RunPod *is currently powered off* per the project state at this handoff time; the resumption host does not need to use RunPod.

## Non-negotiables (do not skip, do not relax)

These are the boundaries the original analysis (`docs/PAPER_GRADE_ANALYSIS_20260507.md` on PR #3) and verification report (`docs/PR3_VERIFICATION_REPORT_20260507.md`) established:

1. **DO NOT** retrain at the reduced PoC config (1000+1000 / max_length=128 / epochs=1). It does not produce a usable AUC and it nearly hung anyway.
2. **DO NOT** use Anthropic Claude as a judge in this domain. Empirically broken (acc=0.50). The whole point of P1-B is to replace it.
3. **DO NOT** commit the `runs/` checkpoint directory. The model is reproducible from base + classifier head; checkpoints are too large for git.
4. **DO** keep the entropy gate on `clean_ad_spam.py` at default `0.0` (disabled). Empirical real-corpus calibration in `.planning/calibration/` shows the threshold needs to be ≥5.5 to drop anything; lower values (3.8 was tried) drop nothing on real data.
5. **DO** verify `style_classifier_auc` is added to `phase6_eval.py`'s `GATE` dict with operator `"le"` and threshold `0.65`. Lower AUC = AI matches the community distribution = good (gate passes).
